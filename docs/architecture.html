The one rule that prevents chaos

n8n only touches events, scheduling, notifications, and human approval delivery. It never makes a content or brand decision — that always stays in the LangGraph brain. Break this rule and the system becomes untestable.

Do we need LangGraph if we have n8n?

Short answer: yes. They solve different problems.

Why separate sub-agents instead of n8n alone?

  • Different model per job — cheap Claude Haiku for caption drafts, reasoning Claude Sonnet for strategy. One mega-prompt would be unreliable and expensive.
  • Least-privilege tools — only the Publishing node can publish; only Engagement can reply to DMs. A single agent holding every capability is a security risk.
  • Focused + testable — each agent has a narrow contract you can unit-test. n8n chaining LLM nodes gives you none of these guarantees.

You could drop LangGraph only if the agent were a single dumb "write a caption" call. This one has validation loops, confidence gating, an agentic classifier, and human interrupts — that's LangGraph's job.

Who comes up with the idea?

The Strategy agent invents the plan; the Creative agent executes a single item from it.

brand profile + campaign brief + (future) past performance │ ▼ Claude Sonnet ──► structured 7-day calendar │ ▼ [ {day:0, pillar:"product spotlight", format:"feed_post", topic:"Achari Momo close-up", variety:"Achari"}, {day:1, pillar:"community", ...}, ... ] ← each entry = one "idea"

How it generates ideas

  • It is bounded by the brand's content pillars (product spotlight 3×/wk, behind-the-scenes 1×, community 1×, offer as needed) and the 31-variety menu.
  • Claude Sonnet returns a typed ContentCalendar (list of ScheduledPost), so the output is structured, not free text.
  • Each calendar entry becomes a PostBrief handed to Creative.

Honest limitation (today)

Idea generation is currently fairly mechanical — a calendar derived from the brief. A richer "idea engine" (seasonal monsoon framing, local Wagholi events, performance-weighted past winners) is the learning-loop upgrade and isn't wired yet, because performance feedback (embeddings + store_post) is still a placeholder.

Why the image must be uploaded

Instagram's API never takes raw image bytes — it fetches a public URL server-side. So every generated image is uploaded to a public Supabase Storage bucket first, and that URL is what Publishing sends.

What is the Validator, and why do we need it?

LLMs drift. The Validator is the deterministic guarantee that whatever the model writes either meets brand rules or gets flagged.

How it works

The caption JSON is parsed into a Pydantic Caption model whose validators hard-check:

  • Hashtags: 8–12 total, all 3 core tags present, no banned tags
  • No banned phrases ("best momos", "you deserve", hyperbole…)
  • Caption length within range; confidence is a real 0–1 number
Claude draft ──► Pydantic validate │ ┌───────────┴───────────┐ PASS FAIL │ │ ▼ ▼ keep caption inject error into prompt, retry (repair loop, ≤2×) │ still failing? ▼ confidence = 0 → human review

Confidence router

Every output carries a confidence_score. Below 0.7 sets human_review_required — the post can't auto-proceed. This (plus the eval harness in CI) turns probabilistic LLM output into a reliable, brand-safe contract.

Why needed: without it you'd eventually publish "BEST MOMOS IN PUNE!!! 🔥🔥" with 20 banned hashtags. The validator makes that structurally impossible.

How the Engagement agent works

The one true agentic loop — it decides per message what to do, rather than following a fixed transform.

new comment / DM (via webhook) │ ▼ classify with Claude Haiku → intent (question / compliment / complaint / booking / spam) → suggested reply → confidence → escalate? (keywords: refund, catering, legal, food poisoning…) │ ┌────┴───────────────┬────────────────────┐ ▼ ▼ ▼ safe + confident escalate OR conf<0.7 DM window expired │ │ │ ▼ ▼ ▼ auto-reply send to human discard, never retry

Compliance baked in

  • Reply to DMs only within the 24-hour user-initiated window; never initiate a DM.
  • On DM_WINDOW_EXPIRED → discard the reply (never retry/queue).
  • Comment-reply and DM-reply tools are scoped to this node only.

built the classifier + escalation logic. missing live webhook ingestion (n8n) — today it processes a queue passed in state, not a live comment feed.
